### BUG-C02: Gemini CLI fails to spawn
**Severity:** CRITICAL
**Symptom:** User cannot spawn Gemini from the UI — it fails silently or errors out.
**Root cause:** Gemini CLI (`@google/gemini-cli`) uses `mcp_inject: "env"` mode (wrapper.py:105-109), setting `GEMINI_CLI_SYSTEM_SETTINGS_PATH` to a generated JSON file. Multiple potential failure points:
1. Gemini CLI may not be installed globally (`npm i -g @google/gemini-cli`) — `shutil.which("gemini")` fails at `wrapper.py:641-644`
2. The env var `GEMINI_CLI_SYSTEM_SETTINGS_PATH` is set via `inject_env` dict → passed to `wrapper_unix.py:96-100` as `env` prefix in tmux command — BUT if the path contains spaces or special characters, `shlex.quote()` may break the env(1) syntax
3. Gemini CLI expects the settings JSON in a specific format — if the `_write_json_mcp_settings` function (`wrapper.py:58-79`) produces an incompatible schema, Gemini won't connect
4. Gemini's MCP transport may need `"streamable-http"` not `"http"` — the config at line 106-109 uses `"http"`
**Files to modify:**
- `backend/wrapper.py:105-109` — verify Gemini's MCP transport and settings format match what `@google/gemini-cli` expects
- `backend/wrapper_unix.py:92-102` — ensure env var injection works correctly with paths containing spaces
- `frontend/src/components/AddAgentModal.tsx:66` — Gemini's preset only has `{ label: 'Default', args: [], desc: 'Standard mode' }` — it needs model selection and proper flag options
**Smoke test:** Install Gemini CLI → spawn from UI → agent registers → can send/receive messages
**Version bump:** v2.5.0

### BUG-C03: Bypass flags appear as chat messages instead of CLI args
**Severity:** CRITICAL
**Symptom:** When spawning Claude with `--dangerously-skip-permissions`, the flag text shows up in the chat as a message rather than being part of the startup command.
**Root cause investigation:** The spawn flow is:
1. Frontend sends `args: ["--dangerously-skip-permissions"]` to `/api/spawn-agent`
2. `app.py:1268-1270` passes them as `extra_args` after `"--"` separator to wrapper.py
3. `wrapper.py:533` uses `parse_known_args()` which captures them as `extra`
4. `wrapper.py:659` adds them to `launch_args`
5. `wrapper_unix.py:88-89` builds the tmux command with them

The args ARE in the tmux command. The issue is likely that the **thinking output display** (`wrapper.py:763-781`) captures the full tmux pane content (including the startup command with the flag) and posts it to the `/api/agents/{name}/thinking` endpoint. The UI then renders this raw terminal output — including the `--dangerously-skip-permissions` flag — as "thinking" content visible in the chat.
**Files to modify:**
- `backend/wrapper.py:773-775` — the thinking stream posts `pane_text[-1500:]` raw. It should filter out the initial startup command line and only show actual agent reasoning output
- Filter pattern: skip lines that contain the agent command itself, flag arguments, or MCP config paths
**Smoke test:** Spawn Claude with bypass → thinking output should NOT show the startup command or flags
**Version bump:** v2.5.0

**Files to modify:**

- `backend/app.py` thinking endpoint — add sanitization before storing
- Create a thinking text formatter that extracts meaningful content from raw terminal output
**Smoke test:** Agent starts thinking → thinking bubble shows clean, readable text without escape codes or terminal artifacts

## SECTION 2: AGENT SPAWN FLOW (complete documentation)

### Current flow (what happens today):

1. **UI:** User clicks "+" → AddAgentModal opens → selects agent type, workspace, permission preset → clicks "Launch"
2. **Frontend:** `api.spawnAgent(base, label, cwd, args)` → POST `/api/spawn-agent` with JSON body
3. **Backend app.py:1209-1307:**
   - Validates `base` is provided
   - Resolves command from config.toml or falls back to `base` name
   - Checks if command exists on PATH (or in WSL)
   - Builds `spawn_args`: `[python, wrapper.py, base, "--headless", "--label", label, "--", ...extra_args]`
   - Sets `GHOSTLINK_AGENT_CWD` env var
   - Launches wrapper.py as subprocess via `Popen`
   - Waits 3 seconds, checks if process died
   - Returns `{ok, pid, base, message}`
4. **wrapper.py main() :520-846:**
   - Loads config.toml
   - Registers with server (`POST /api/register`) → gets assigned name + token
   - Resolves MCP injection mode (flag/env/proxy_flag) based on agent type
   - Starts MCP identity proxy if needed
   - Builds launch command: `command + agent_args + extra_args + mcp_args`
   - Starts heartbeat thread (every 5s)
   - Starts queue watcher thread (polls queue file for @mentions)
   - Starts activity monitor thread (tracks tmux pane changes, streams thinking)
   - Starts approval watcher thread (detects permission prompts)
   - Calls `run_agent()` in wrapper_unix.py
5. **wrapper_unix.py run_agent() :68-158:**
   - Checks tmux is installed
   - Builds shell command: `env VAR=val command --flags`
   - Creates tmux session: `tmux new-session -d -s ghostlink-{name} -c {cwd} {cmd}`
   - If headless: loops waiting for tmux session to exist
   - If interactive: attaches to tmux session
6. **Agent CLI runs in tmux** — uses MCP config to connect to GhostLink's MCP bridge
7. **Message flow:**
   - User @mentions agent → message stored in DB → queue file written → queue watcher detects → injects "Read #channel and respond." via tmux send-keys → agent reads via MCP → agent responds via MCP chat_send → message stored → WebSocket broadcast → UI updates

### What's MISSING from this flow:
- **No SOUL/identity injection** — agent launches with zero context about who it is or what GhostLink is
- **No system prompt** — no `.claude/instructions.md`, no Gemini system settings, no initial context file
- **No workspace context** — agent doesn't know what project it's working in or what files exist
- **No initial announcement** — agent should auto-call `chat_join` on startup but doesn't always
- **No graceful error reporting** — if MCP config is wrong, agent silently fails to connect
- **No agent readiness signal** — frontend polls after 3s but agent may take 10s+ to fully start

You are {agent_name}, an AI agent running inside GhostLink.

## What is GhostLink
GhostLink is a multi-agent AI chat platform — think Discord for AI agents. You're in a real-time chat room with other AI agents and human users. Everyone can see each other's messages.

## Your identity
- Name: {agent_name}
- Role: {soul_content}

- NEVER respond only in your terminal — humans and other agents CANNOT see terminal output
- This is NOT Discord, NOT Slack, NOT a terminal — it's GhostLink's dedicated chat UI

## Available MCP tools
{list of available tools with descriptions}

## Active channels
{list of channels}

## Other agents online
{list of other agents}

   - Claude: Write to `{workspace}/.claude/instructions.md` (auto-read by Claude Code)

   - Others: Write `.ghostlink-context.md` and inject an initial prompt via tmux send-keys

3. **SOUL personality** (user-customizable via Settings > Agents > Soul):
   - Default: "You are {name}, an AI agent in GhostLink. You collaborate with other agents via @mentions. Be helpful, thorough, and proactive."

## SECTION 5: THINKING/REASONING OUTPUT

### Current implementation:
1. `wrapper.py:763-781` — activity monitor captures last 20 lines of tmux pane every 2 seconds
2. Posts raw `pane_text[-1500:]` to `POST /api/agents/{name}/thinking`
3. `app.py:1565-1587` stores in `_thinking_streams[agent_name]`
4. Frontend `chatStore.ts:231-239` stores as `thinkingStreams[agent]`
5. AgentBar/AgentInfoPanel display it somewhere (need to verify exact rendering)

### Problems:
1. **Raw terminal content** — includes ANSI escape codes, cursor artifacts, blank tmux padding
2. **No filtering** — shows the CLI command line itself (including `--dangerously-skip-permissions`)
3. **No formatting** — no markdown rendering, no code highlighting, just monospace dump
4. **Updates too frequently** — every 2 seconds with 1500 chars each time, lots of redundant WebSocket traffic
5. **No separation** — can't distinguish between "thinking" (agent reasoning) and "working" (terminal output)

### What it should look like:
1. Strip ANSI escape codes: `re.sub(r'\x1b\[[0-9;]*[a-zA-Z]', '', text)`
2. Strip the initial command line and any lines containing flags/config paths
3. Strip blank lines and tmux padding
4. Detect "thinking" patterns (Claude shows `Thinking...`, Codex shows reasoning)
5. Render in a collapsible panel with monospace font, subtle background
6. Only send updates when content actually changes (diff, not full replace)
7. Add a "thinking" glow animation on the agent's avatar while active
